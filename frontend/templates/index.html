§ 4 — the nine layers

What surrounds the model.

The reliability of an LLM-powered system is not the model. It is the harness around the model. Nine layers, each catching a different class of failure. Every file in the repo maps to one of these layers — or is documented as cross-cutting infrastructure.

Read from L1 up: the agent's instructions sit at the bottom; everything above it is an additional layer of safety, validation, or measurement that the request must pass through. Each band lists the actual components — every one of them has a file in this repo. The dashed strip at the bottom is the external infrastructure the layers depend on.

§ 4b — message → decision

How a single chat message becomes a refund decision.

The static stack above shows what exists. This is what runs. Every customer message walks the same path through the LangGraph state machine — eight nodes, deterministic transitions, every node observable. The output is the DECISION · KIND · $AMOUNT · clauses: … chip you see under each reply in the chat.

The eight nodes are deterministic, not LLM-driven. Only classify_intent and respond actually call the LLM — every other step is a typed tool call or a graph transition. Every node emits a LayerEvent the SSE bus picks up, so each chat turn is a fully observable timeline in /admin → Live Trace.
